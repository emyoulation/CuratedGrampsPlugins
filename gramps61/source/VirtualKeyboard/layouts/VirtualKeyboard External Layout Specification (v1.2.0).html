VirtualKeyboard External Layout Specification

Version 1.2.0

This document defines the format, semantics, and validation rules for external keyboard layout files used by the VirtualKeyboard Gramplet for Gramps.

The primary goals are:

  • Human-editable layout definitions
  • No Python knowledge required
  • Predictable loading and failure behavior
  • Safe extensibility without cross-file coupling

This specification is normative for VirtualKeyboard ≥ 1.2.0.

1. File format and location

  • External layouts are defined in CSV files
  • Files are stored in a layouts/ subdirectory relative to virtualkeyboard.py
  • Each CSV file defines exactly one layout set
  • Files are loaded independently; file order does not matter
  • External layout CSV files must be encoded in UTF-8 (without BOM)

Files using other encodings may fail to load or produce incorrect characters. This requirement ensures correct handling of diacritics, non-Latin scripts, and emoji.

2. Comment handling

  • Lines whose first non-whitespace character is # are comments
  • Comment lines are ignored by the loader
  • Comments may appear anywhere in the file

3. Directive syntax

All non-comment content is expressed as directives.

@key,value
  • @key is case-insensitive
  • Everything after the first comma is treated as the value
  • Unknown keys are ignored (forward compatibility)

4. Layout set header (required)

Each file must begin with exactly one layout-set header introduced by @set.

Required directives

KeyDescription
@idUnique identifier for the layout set
@languagePrimary language code (ISO-639-1 / BCP-47)
@buttonButton label or icon (e.g. emoji flag)
@default_layoutLayout ID selected by default

Optional directives

KeyDescription
@language_iso3ISO-639-3 language code
@regionRegion code (e.g. FR, DE)
@label_nativeLayout-set name in native language
@label_enLayout-set name in English
@tooltip_nativeTooltip in native language
@tooltip_enTooltip in English

5. Layout blocks

One or more layout blocks must follow the layout-set header.

Each block begins with @layout.

Required directives

KeyDescription
@idLayout identifier (unique within the set)
@labelButton label shown in the UI

Optional directives

KeyDescription
@label_enEnglish fallback label
@tooltip_nativeTooltip in native language
@tooltip_enTooltip in English

6. Layout ID rules (important)

  • Layout IDs are case-insensitive
  • During import, all layout IDs are normalized to UPPERCASE
  • Authors should write IDs in uppercase
  • _SHIFT, _ALTGR, _ALTGR_SHIFT, etc. are conventional suffixes
  • UI presentation is controlled by labels, not IDs

This rule exists to avoid case-sensitivity problems on platforms such as Windows.

7. Row definitions

Keyboard rows are defined using the @row directive.

@row,"<characters>"

Rules:

  • @row directives may only appear inside a layout block
  • Each @row defines exactly one keyboard row
  • The value must be a quoted CSV string
  • Anonymous row lines are invalid
  • Row order is significant

8. Quoted string escaping

All quoted string values follow standard CSV escaping rules:

  • A double quote character (") inside a quoted string is written as ""
  • No other escape sequences are defined
  • Backslashes (\) have no special meaning and are treated literally

The loader relies on standard CSV parsing; no custom escaping is implemented.

9. Row mapping resolution and precedence

When a layout references a row mapping name:

Resolution order:

  1. Row mappings defined within the same CSV file
  2. Built-in VirtualKeyboard row mappings
  3. Failure if unresolved

Local definitions override built-ins. External files never reference other external files.

10. Isolation rule (no inheritance)

  • Each CSV file is fully self-contained
  • No external file may inherit from or depend on another
  • This rule is intentional and absolute

11. Validation and failure behavior

  • Layout sets are validated as a whole
  • All validation errors are reported together
  • If any error occurs:
    • The entire layout set is rejected
    • Warnings are emitted to the Gramps status bar and console
    • VirtualKeyboard falls back to a built-in "Oops" layout set
  • Partial loading is not permitted

12. Language selection behavior

  • External layout files provide their own translations
  • gettext/Weblate is not used for external layouts
  • At runtime:
    • Native language strings are preferred when the UI language matches @language
    • English strings are used as fallback
    • If only one is present, it is used unconditionally

13. Reference implementation

The file:

layouts/fr.csv

is the normative example for this specification.

It demonstrates:

  • Proper header structure
  • Correct AZERTY layout definition
  • Use of @row directives
  • Mandatory quoting and CSV escaping
  • Layout ID normalization
  • Native and English labels/tooltips
  • Reliance on built-in rows where appropriate

Implementations and contributors should treat fr.csv as the canonical model.

14. Design principles (non-normative)

  • Human readability over cleverness
  • Explicit over implicit
  • Predictable failure over silent fallback
  • No cross-file coupling
  • Extensible without breaking compatibility

Status:
This specification is frozen for VirtualKeyboard 1.2.0.

Future revisions may extend the format but must not invalidate compliant files.
